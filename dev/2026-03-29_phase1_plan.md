# Phase 1: File Management & Dirty Tracking

## Context

Galley is at MVP/v1 with all 6 original epics complete. Phase 1 of the roadmap adds three features — Download, Upload, and Unsaved Changes Tracking — that are foundational for subsequent phases. Dirty tracking (1c) is especially critical as a prerequisite for Phase 2's conflict detection, auto-reload, and undo.

## Implementation Order

**1a Download → 1b Upload → 1c Unsaved Changes Tracking**

Download is simplest and introduces the pattern for the new `/download` route. Upload reuses the same backup+write logic (motivating a shared helper). Dirty tracking is purely client-side and touches no server code, making it a clean final step.

## 1a. Download

### Server: `src/app.js`

**Extract a `validateFilename` helper** to DRY up the 3-layer path validation currently duplicated in `/edit/:filename` (lines 74-86) and `/save/:filename` (lines 102-114). This helper returns `{ filePath }` on success or `{ error, status }` on failure. Refactor both existing routes to use it.

**Add `GET /download/:filename`** (after `/edit/:filename`, before `/save/:filename`):
- Uses `validateFilename` for path security
- Checks file exists with `access()`

- Uses `res.type('html').send(html)`

**Update file picker HTML** (GET `/`):
- Add a third column header: `<th></th>` (no label, keeps it minimal)
- Add download link in each row: `<a href="/download/...">Download</a>`
- Update empty state colspan from 2 to 3
- Add subtle styling for download links

### Client: `src/galley-client.js`

**Add download link** in the `DOMContentLoaded` handler (after save button creation):
- Create an `<a>` element with `id="galley-download"`, `href="/download/<filename>"`
- Append to `#galley-ui` container (or body fallback), same pattern as save button

### Styles: `src/galley-styles.css`

- Add `#galley-download` styles: fixed bottom-right, offset left of save button (~7.5rem right), outlined variant of save button style (white bg, blue border/text)

### Tests: `tests/app.test.js`

## 1b. Upload

### Approach: JSON POST (no new dependencies)

Instead of multipart/form-data (which would require `multer`), the file picker page reads the file client-side with `FileReader.readAsText()` and POSTs JSON `{ filename, html }` — same pattern as the existing save endpoint. This keeps zero new production dependencies.

### Server: `src/app.js`

**Extract an `atomicWriteWithBackup` helper** from the save route's backup+write logic (lines 126-142). Parameters: `(filePath, content, resolvedBackupDir, { requireExists })`. The save route sets `requireExists: true` (404 if missing); the upload route sets `requireExists: false` (skip backup if new file).

**Add `POST /upload`**:
- Expects JSON: `{ filename: "name.html", html: "..." }`
- Validates `filename`: required string, `.html` extension, no `..`/`/`/`\`, resolved path within docsDir
- Validates `html`: required non-empty string
- Calls `atomicWriteWithBackup` with `requireExists: false`
- Returns `{ ok: true }` on success

### File picker update in `GET /`

Add an upload section below the table:
```html
<div class="upload-section">
  <label for="galley-upload">Upload HTML file</label>
  <input type="file" id="galley-upload" accept=".html">
</div>
```

Add inline `<script>` on the file picker page:
- Listen for `change` on `#galley-upload`
- Read file with `FileReader.readAsText()`
- POST to `/upload` with `{ filename: file.name, html: reader.result }`
- On success: `window.location.href = '/edit/' + encodeURIComponent(filename)`
- On error: `alert()` with server error message

Add inline CSS for `.upload-section` (margin, subtle styling).

### Tests: `tests/app.test.js`

New `describe('POST /upload')` (uses temp dir pattern from save tests):
- Uploads a new HTML file and verifies it's written to disk
- GET `/edit/:filename` works after upload
- Creates backup when overwriting existing file
- Rejects non-HTML filename
- Rejects path traversal
- Rejects missing/empty html or filename fields
- File picker page includes upload input

## 1c. Unsaved Changes Tracking

### Client: `src/galley-client.js`

**New state** alongside existing `saving` var (line 67):
- `var dirty = false;`

**New `setDirty(value)` function:**
- Sets `dirty = value`
- Toggles `galley-dirty` class on save button
- Updates `document.title`: prepend `\u2022 ` (bullet) when dirty, remove when clean (store original title at load)

**Delegated `input` listener** on `document`:
- Check `e.target.closest('[contenteditable="true"]')`
- If match, call `setDirty(true)`

**`beforeunload` handler** on `window`:
- If `dirty`, call `e.preventDefault()` and set `e.returnValue = ''`

**Integration with save flow** (in `xhr.onload` success branch, line 176-177):
- After `showToast('Saved')`, call `setDirty(false)`

### Styles: `src/galley-styles.css`

Add `position: relative` to existing `#galley-save` rule (needed for `::after` positioning).

Add dirty indicator dot:
```css
#galley-save.galley-dirty::after {
  content: '';
  position: absolute;
  top: -3px;
  right: -3px;
  width: 8px;
  height: 8px;
  background: #e67e22;
  border-radius: 50%;
  border: 2px solid #fff;
}
```

### Tests: `tests/galley-client.test.js`

New `describe('unsaved changes tracking')`:
- Sets dirty flag on `input` event in editable element (dispatch `new Event('input', { bubbles: true })`)
- Does not set dirty on input in non-editable element
- Updates document title with bullet prefix when dirty
- Save button gets `galley-dirty` class when dirty
- Clears dirty indicator after save (test `setDirty(false)` path)
- Registers `beforeunload` handler (spy on `window.addEventListener`)
